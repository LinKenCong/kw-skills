# 周报输出格式

最终输出必须是中文 Markdown。每个日期范围生成一个文件。正文只写有记录的项目，不保留空项目小节，也不提及无记录项目。

## 文件模板

```markdown
# 工作周报

**报告周期**: YYYY-MM-DD ~ YYYY-MM-DD
**报告人**: [Git username]
**项目范围**: [有记录的项目名列表]

---

## 本期概览

- 提交次数：XX 次
- 涉及项目：X 个
- 主要模块：[模块 A]、[模块 B]、[模块 C]

---

## 主要工作内容

### [项目 A]

#### [工作主题一]

- 完成 [具体能力/模块] 的开发或调整，涉及 [关键文件/模块]
- 处理 [问题/目标]，改动集中在 [目录/组件]
- 相关提交：`abc1234`、`def5678`

#### [工作主题二]

- 优化 [流程/接口/配置]，提升 [可维护性/稳定性/可用性]
- 相关提交：`ghi9012`

### [项目 B]

#### [工作主题三]

- 完成 [具体能力/模块] 的开发或调整，涉及 [关键文件/模块]
- 相关提交：`ghi9012`

---

## 数据说明

- 数据来源：Git 提交记录，按已确认的 Git 作者过滤条件过滤
- 异常跳过项目（仅在存在非 Git、缺失路径或无法读取 Git 记录时写入；没有异常时整条省略；无记录项目不要列入）：
  - `/path/to/not-git`：不是 Git 仓库
```

## 写作规则

- 不按日期拆分工作内容。
- 不写“周一做了什么”“某天完成什么”。
- 不写工作量评估。
- 不写趋势分析。
- 不和上周、上月、历史周期对比。
- 不给无记录项目创建空小节，也不要在数据说明或跳过项目里列出无记录项目。
- 不输出“新增 N 行、删除 N 行、文件改动 N 项”等代码变更统计。
- `## 主要工作内容` 必须先按项目分组：项目名用 `###`，项目内工作主题用 `####`。
- 可保留 commit hash 作为依据。
- 只有 Git 数据支持的内容才能写进报告。

## 敏感信息处理

- 报告人使用 Git username（`git config user.name`），不要写邮箱。
- 默认文件名不要包含完整邮箱、token、密钥、远程仓库凭据或本机用户主目录。
- 报告正文只基于 Git 元数据生成，不读取源码文件内容。
- 如果 commit message、分支名或文件路径中出现疑似密钥、token、密码、私钥片段，写成概括描述或 `[已脱敏]`，不要原样复制。
- 绝对路径只用于用户确认过的项目路径和跳过说明；如果用户说明报告会对外分享，优先使用项目名或相对路径。

## 无记录周期

如果某个日期范围所有项目都没有有效记录，不生成空文件。最终回复说明：

```markdown
未生成 `weekly-report_YYYY-MM-DD_YYYY-MM-DD_git-user.md`：该周期内所有项目均无当前作者提交记录。
```
