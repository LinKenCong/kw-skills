# 周报生成器

根据 Git 提交历史和代码变更，生成中文 Markdown 个人工作周报。

## 使用场景

当用户请求以下内容时使用此 skill：
- 生成周报、写周报、本周工作总结
- 汇总这周提交、总结代码变更、导出开发日志
- 为一个或多个项目生成指定周期工作报告

## 核心规则

1. 默认只关注当前 Git 账户。用户没有指定作者时，不要额外询问作者；解析当前 Git 账户后，在最终确认清单中展示。
2. 必须确认项目路径。用户没给项目路径时，询问使用当前路径项目，还是指定一个或多个项目路径。
3. 用户没给日期范围时，询问默认本周内，还是指定一个或多个日期范围。
4. 必须确认输出目录。默认建议 `.scratch/weekly-report/`，但要让用户确认或改写。
5. 收集完项目、日期、作者、输出目录、计划文件名后，必须先让用户确认；确认前不要读取 `git log`，不要生成文件。
6. 每个日期范围生成一个 Markdown 文件；多个项目写进同一周期文件，按项目分节。
7. 指定时间范围内没有工作记录的项目要跳过，不要在正文留下空项目小节。
8. 非 Git 项目、缺失路径、无法读取 Git 记录的项目要跳过，并在最终回复和报告的数据说明中写明。
9. 不按具体日期分类，不写“某天产出某工作”。按工作主题、模块、项目组织内容。
10. 不做工作量评估，不做趋势分析，不和历史周期对比。
11. 优先使用 `scripts/collect-git-activity.sh` 收集 Git 数据；脚本不可用或失败时，改用手动 Git 命令兜底。
12. 默认文件名只是建议，不是固定输出规则；最终确认清单必须展示计划文件名，并允许用户改名。
13. 生成报告时避免泄露不必要的敏感信息：文件名不要默认包含邮箱；正文只使用 Git 元数据，不读取源码内容；commit message 或路径中出现疑似密钥、token、密码时要脱敏或概括。

## 示例对话

**用户**: 帮我生成本周周报

**助手**: 请先确认项目路径：
1. 使用当前路径项目 `/path/to/current`
2. 指定一个或多个项目路径

**用户**: 用当前项目

**助手**: 请确认日期范围：
1. 默认本周内：2026-07-06 ~ 2026-07-10
2. 指定一个或多个日期范围

**用户**: 默认本周

**助手**: 请确认输出目录：
1. 使用 `.scratch/weekly-report/`
2. 指定其他目录

**用户**: 默认目录

**助手**: 准备生成周报，请确认：

```markdown
- Git 账户：bob <bob@example.com>
- 时间范围：2026-07-06 ~ 2026-07-10
- 项目路径：/path/to/current
- 输出目录：.scratch/weekly-report/
- 计划文件：weekly-report_2026-07-06_2026-07-10_bob.md

确认后才会读取 git 记录并生成文件。
```

用户确认后，按 [执行流程](./references/execution-flow.md) 读取 Git 数据并生成周报。
